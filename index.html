專案背景與目標

整合多協定的智能分析平台，透過 AI 自動化解析封包、生成測試案例、Mantis Bug AI 辨識與 n8n Workflow 自動化，大幅提升測試效率與準確度。

🔴 傳統測試痛點

  • 🔀多協定複雜度高
    OMCI、WAN、SIP、ACS、WiFi 各有不同封包格式與流程規範，人工解析耗時費力。
  • 📝測試用例維護困難
    數百個測試場景需人工編寫與維護，容易出錯。
  • 🔍異常點偵測不完整
    依賴測試人員經驗，容易遺漏邊界條件與隱蔽的異常情況。
  • ⏱️測試效率低落
    從封包擷取到報告生成，需多個工具串接，流程繁瑣且耗時。
  • 🔄跨版本比對困難
    不同 PCAP 版本間的差異分析，需人工逐一比對，難以快速識別差異。

🎯 主要目標

將多協定封包轉換為語意 Token，結合規則引擎與 AI 推論，實現自動化分析、差異比對與測試案例生成。

  • Spec-driven Shift-Left — 設計階段即生成測試案例 (PDF/Docx/Excel)
  • 統一多協定平台 — Spec + OMCI、WAN、SIP、ACS、WiFi + WEB + Mantis Bug + n8n
  • AI 自動解析封包內容 — 自動生成測試用例與驗證建議，減少人工介入
  • 異常點偵測 — 從封包擷取到報告生成的全自動化流程
  • 不同版本韌體 PCAP 行為比對差異 — 快速識別跨版本變更
  • 自動上傳 TestRail — AI 生成測試案例直接整合至測試管理系統
  • Mantis Bug AI 辨識 — 上傳 封包/Log/圖片/影片，AI 自動撰寫 Bug 並一鍵上傳 Mantis
  • n8n Workflow 自動化 — AI 分析報告透過 Webhook 自動寄送到Jenkins和信箱

Stage 0：Raw Inputs

系統支援多源資料的統一接收。在此階段，資料保持原始格式，等待被處理。

PCAP / PCAPNG Console log / Hex HTTP / XML CSV / Excel Log / Image / Video (Mantis)

這一層還沒有 token，只有 raw data。Mantis Bug 模組接收 Log/圖片/影片作為輸入。

Stage 1：Token-based (Parse and Decode)

"Turning Everything into Tokens" — 將所有原始資料轉換為結構化語意 Token。

以 OMCI Token Generator 為例，主要函式包含 extract_omci_packets() 與 parse_omci_packet(...)，產生結構化的 tokens：

{
  "ME_class": 171,
  "ME_name": "Extended VLAN Tagging Operation",
  "Attribute_List": {...},
  "VLAN_tree": {...},
  "bit_fields": {...}
}

每個 Attribute / 每個 VLAN rule / 每個 bit flag 全部都是語意 token。

Stage 2：Rule-based Analysis Layer

這一層是 Rule-based 確定性 (Deterministic Layer) 的核心。不依賴 AI，而是使用工程等級的規則引擎，對 Token 進行快速、精確的判斷。

🚫
No AI
完全基於預定義規則，沒有幻覺，沒有不確定性
⚡
Fast
極速執行，適合即時封包分析與處理
🔁
Repeatable
相同輸入永遠產生相同輸出，便於除錯
🔧
Engineering Grade
精確判斷機制，非模糊的結果推測

範例：WiFi MLO Rule Engine

def mlo_flow_result(self, flow):
    if not flow["assoc"]:
        return "FAIL"
    if not all([flow["eapol_m1"], ...]):
        return "FAIL"
    if not (flow["data_2g"] or flow["data_5g"] 
        or flow["data_6g"]):
        return "FAIL"
    return "PASS"

flow = token, rule = deterministic reasoning. PCAP Compare 中的 packet count compare、band distribution compare、behavior diff 全部是 token → rule → result。

Stage 3：LLM AI 語意推論 (OpenAI)

AI-Driven Semantic Inference — 給 AI 的是 Decode 後的語意 Token，而不是 Raw Hex。

1. 單一 Token → AI 解釋

將 Decode 後的語意 Token 送入 AI，獲得專業解釋。

2. Summary (Token Sampling)

大型 PCAP 檔案包含數萬個封包。只選擇具代表性的關鍵 Token 進行 AI 分析，避免 Context Window 溢出。

3. Compare PCAP Behaviors

AI 比較 token 統計、token 行為差異、token 分布，進行深度語意分析與異常檢測。

4. Test Case Generation

token → AI logic → TestRail case，從結構化資訊自動生成測試案例。

Stage 4：Output & Integration Layer

Turning Tokens into Tangible Assets — 將 Token 轉化為有形資產，並整合 Mantis Bug Tracker 與 n8n Workflow 自動化。

問題及解決方案

在開發過程中遇到的關鍵技術挑戰與對應的解決策略。

🔴 LLM Token 截斷問題

Wi-Fi Test Cases 生成不穩定的根本原因分析 — 問題現象：Wi-Fi Test Cases 生成時好時壞，非隨機 Bug。報錯訊息：[WiFi AI] JSON parse failed，JSON 結構在結尾處中斷 (Truncated)。

Root Cause：Context 過大導致觸及模型 Token 上限。WiFi 模組的 Prompt 包含大量 WiFi Decode 資料 (Radiotap + Management frames)，要求一次生成 10 Test Cases × 5 Steps，輸出量大。Token 滿載時，JSON 尾部被切斷。

為何 WAN/SIP 比較穩定？

項目WiFiWAN / SIP
Decode 資料長度極長 (Radiotap + 802.11)短 (IP/SIP header)
Prompt Token 消耗高 (接近上限)中

三層防護解決方案

🛡️ 第一層防護（最小改動）

策略 1：限制 Raw Context 長度，設定 MAX_WIFI_CONTEXT_LEN = 6000，效益：解決 70% 的 JSON parse failed。
策略 2：實作 Retry 機制，JSON parse 失敗時自動重試一次 (Prompt 不變)，結合策略 1 穩定度可達 95%。

🛡️ 第二層防護（降低 Token 消耗）

核心觀念：WiFi Decode 對「生成測項」是雜訊。完全移除 Prompt 中的 WiFi Decode (Radiotap/Mgmt)，改下拉選單 (Category/Focus) 生成。效益：Token 使用量降低 60~80%，徹底解決截斷問題，AI 更聚焦測試邏輯。

🛡️ 第三層防護 (Safe Parsing)

修正 JSON 格式，寫入 safe_json_loads()。這是 Production 級別 LLM 系統必備工法。功能：修正字串、引號、移除多餘逗號後再 Parse。問題延伸：Security 類測項的特殊符號 (如 AES-CCMP)、Trailing comma、智慧引號、未跳脫雙引號等。

總結：限制 Context 長度 + 拿掉 WiFi Decode + Safe Parser = 100% 穩定

⚡ 效能優化與系統限制

Performance Optimization

  • Multi-threading Architecture：使用 threading 將耗時處理移至背景，確保 GUI 在大型檔案下仍維持流暢
  • Smart Packet Filtering：透過 pyshark display filters 僅擷取相關協定封包，降低記憶體與處理時間
  • Efficient Data Export：採用 pandas DataFrame 批次處理，顯著提升 Excel 報表生成速度
  • Non-blocking 非阻塞流程：ai_button_busy / restore 機制確保 UI 響應

Limitations & Safety

  • Recommended File Size：建議單檔 PCAP 控制在 50MB 以內，以避免解析延遲或記憶體問題
  • AI Context Window：受限於 LLM token，建議單次分析不超過 10,000 封包，上下文會自動截斷
  • 摘要化資料送 LLM：大型 PCAP 檔案只選擇具代表性的關鍵 Token 進行 AI 分析
  • pyshark layer 存取：只解析必要封包，降低資源消耗

WAN 網路測試分類架構

IPv4 與 IPv6 測試基準

IPv4 Stack

1. Physical & Link Layer ▼
  • Link Detection: Up/Down detection, Rapid Link Flap handling
  • Auto-Negotiation: Speed/Duplex fallback (1G/100M), Mismatch handling
  • VLAN Tagging: Accept/Drop untagged frames, VLAN ID matching
  • MAC Handling: Connectivity after MAC address change
2. IPoE (DHCP Client) ▼
  • Basic Operation: DORA process (Discover-Offer-Request-Ack)
  • Lease Handling: Renewal (T1), Rebinding (T2), Expiry behavior
  • Options Verification: Subnet (1), Gateway (3), DNS (6), Lease (51)
  • Resilience: Handling NAK, Server Unreachable, Retry backoff
3. PPPoE (IPv4) ▼
  • Discovery: PADI → PADO → PADR → PADS sequence
  • Session: LCP negotiation, MRU verification
  • Authentication: PAP / CHAP success & failure scenarios
  • Recovery: PADT handling, Auto-reconnect after link loss
4. Routing & Traffic ▼
  • Route Management: Default route installation/removal on WAN events
  • Connectivity: ICMP Ping to Gateway/Public IP
  • DNS: Resolution validation (Primary/Secondary)
  • Throughput: Stability under load, MTU fragmentation
5. MTU & MSS ▼
  • IPoE MTU: 驗證標準 1500 bytes 傳輸
  • PPPoE MTU: 驗證 1492 bytes (扣除 8 bytes header)
  • TCP MSS Clamping: 確保 TCP 握手時 MSS 自動調整
  • Large Transfer: 大檔案傳輸測試，驗證分片重組能力
6. NAT (IPv4) ▼
  • Basic NAPT: 驗證 Network Address Port Translation 功能
  • Multi-Client: 多個 LAN 客戶端同時存取 Internet
  • Session Mgmt: TCP/UDP NAT session 建立與維護
  • Cleanup: WAN 斷線後自動清除無效 session
7. Stability & Usability ▼
  • Cold Boot: 冷啟動後自動恢復 Internet 連線
  • Repeated Reboot: 連續重啟壓力測試，確保 WAN 不失效
  • Mode Switch: IPoE 與 PPPoE 模式切換的穩定性
  • Config Persistence: WAN 設定儲存與應用驗證

IPv6 Stack

1. Physical & Ethernet ▼
  • Link State: IPv6 stack initialization on Link Up
  • DUID Persistence: DUID consistency across reboots/reconnects
  • MAC Address: Correct WAN MAC usage in IPv6 frames
  • Recovery: Auto-reconnect after cable unplug
2. IPoE (RA / SLAAC) ▼
  • Router Advertisement: Process Prefix, Lifetime, M/O flags
  • SLAAC: Global address generation via EUI-64 or Privacy Ext.
  • DAD: Duplicate Address Detection verification
  • Default Route: Install ::/0 route from RA
3. DHCPv6 & Prefix Delegation ▼
  • Stateful DHCPv6: Solicit → Advertise → Request → Reply
  • IA_NA: WAN interface address assignment
  • IA_PD (MUST): Delegated Prefix acquisition (/56, /60)
  • LAN Integration: Assigning PD subnets to LAN interfaces
4. IPv6 PPPoE ▼
  • IPv6CP: Negotiation of Interface Identifier (IID)
  • Link-Local: Establishment over PPP session
  • DHCPv6 over PPP: PD acquisition over PPPoE tunnel
  • Recovery: Session restoration after WAN interruption
5. MTU & Path MTU ▼
  • Minimum MTU: 確保支援 IPv6 最小 MTU (1280 Bytes)
  • Protocol MTU: IPoE (1500) 與 PPPoE (1492) 驗證
  • PMTU Discovery: 正確處理 ICMPv6 Packet-Too-Big 訊息
  • Large File: 驗證大檔案傳輸的穩定性
6. LAN IPv6 Functionality ▼
  • Address Assignment: LAN 客戶端正確獲取 IPv6 地址
  • Gateway: 正確獲取 IPv6 Default Gateway
  • Connectivity: LAN 客戶端可存取 IPv6 Internet
  • Persistence: WAN 重連後 LAN IPv6 功能自動恢復
7. Stability & Usability ▼
  • Auto-Recovery: 冷啟動後自動恢復 IPv6 連線
  • Stress Test: 反覆重啟不破壞 IPv6 WAN 功能
  • Mode Switch: IPoE/PPPoE 切換驗證
  • Config Save: 設定變更後的存檔與生效驗證
